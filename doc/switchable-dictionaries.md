# Switchable Dictionaries

Most dictionaries in Python programs are string-keyed. Module namespaces,
object and class namespaces, keyword arguments, configuration data, and many
ordinary application mappings use exact strings as keys. When the VM can prove
that a dictionary contains only exact strings, it can use native string hashing
and equality directly instead of calling Python-visible `__hash__`, `__eq__`,
reflected equality, and truth conversion. This removes several calls and inline
cache lookups from common dictionary operations.

Public Python dictionaries must still accept arbitrary hashable keys. CloverVM
therefore implements one public Python class, one C++ `Dict` class, and one
native layout with two operational shapes:

- a canonical string-keyed shape for exact `str` keys
- a general shape for arbitrary Python keys

An exact builtin dictionary starts string-keyed and switches permanently to the
general shape when a non-string key participates in an operation. The shape is
an internal optimization, not a different public mapping type. Switching does
not change object identity, copy storage, or affect insertion order.

See [Dictionaries](dictionaries.md) for a shorter overview and
[Clover C API](clover-c-api.md#dictionary-api) for the native extension
interface.

## Representation And Shapes

A `Dict` contains:

- an insertion-ordered `ValueArray<Entry>`
- an open-addressed `RawArray<int32_t>` probe table
- a live-entry count
- a probe-structure generation

Each live entry stores a key, value, and canonical SMI hash. Deleted entries
remain as invalidated slots in the insertion-ordered array, while their probe
slots become tombstones. Overwriting an existing value does not move the entry.
Inserting a new key appends a new entry.

The two shapes have identical object and storage layouts. The canonical
string-keyed shape proves that every live key is an exact `str`, so an
exact-string operation may use native string hashing and equality without
executing Python. The canonical shapes are cached on `ThreadState` as
`get_exact_dict_string_key_shape()` and
`get_exact_dict_general_shape()`.

The general shape permits arbitrary Python keys. General operations may invoke
`__hash__`, equality, truth conversion, descriptors, and arbitrary Python
bytecode. A string lookup on a general dictionary is still a general operation:
a stored custom key with the same hash may require Python-visible equality.

A semantic operation with a non-string key promotes an exact string-keyed
dictionary before probing. Promotion changes the shape in place and increments
the probe-structure generation, but does not rebuild the table. It preserves
identity and insertion order and is one-way.

Deletion never demotes a dictionary. If hashing or equality raises after
promotion, the dictionary may remain general even when no entry changed.

Only the canonical exact string-keyed shape selects raw string behavior. Other
receiver shapes must use general semantics.

## Hash Normalization

Dictionary entries store canonical SMI hashes. Public `hash()`, exact builtin
hash helpers, and dict storage use the same normalization:

- `False` and `True` become `0` and `1`
- SMI values are preserved
- BigInts that fit in SMI range preserve their integer value
- larger BigInts reduce through CloverVM's Mersenne-form hash modulus
- `-1` is remapped to `-2`
- non-integer `__hash__` results raise `TypeError`

`CanonicalizeHash` owns this protocol-free normalization. General dict bytecode
inlines the public hash protocol as a cache-bearing `__hash__` special-method
call followed by `CanonicalizeHash`. The exact-string path uses
`string_hash_normalized`, which produces the same stored representation.

Equal integer values must therefore have equal hashes across bool, SMI, and
BigInt representations.

## Python Method Dispatch

The public general-key methods are generated with `CodeObjectBuilder` in
`dict.cpp`. Generated bytecode is used for:

- `__getitem__`, `get`, and `__contains__`
- `__setitem__` and `setdefault`
- `__delitem__` and `pop`

These methods contain their own cache-bearing hash and equality sites. Python
protocol execution remains visible to the interpreter and to a future JIT;
trusted opcodes perform only table mechanics.

Special-method operator caches install trusted handlers for exact builtin dicts
when both the receiver and key shapes prove the exact-string case. This covers
subscription lookup, assignment, deletion, and membership without entering the
generated general method.

Regular methods do not currently have the same trusted-handler mechanism.
`setdefault` and `pop` therefore contain explicit string-keyed probes before
their general paths. Other direct or uncached calls into generated methods may
conservatively promote before continuing through general bytecode.

## Trusted Dict Opcodes

The generated methods use dict-specific trusted opcodes:

- `DictPromoteStringKeyed`
- `DictTryStringKeyedSetDefault`
- `DictTryStringKeyedPop`
- `DictProbeStart`
- `DictProbeForLookup`
- `DictProbeForInsert`
- `DictProbeAdvance`
- `DictEntryKey`
- `DictEntryValue`
- `DictEntryStillMatches`
- `DictResizeForInsert`
- `DictInsertNew`
- `DictOverwriteEntry`
- `DictDeleteEntry`

These opcodes are not Python-visible APIs. They may inspect or mutate dict
storage, but they must not invoke hashing, equality, descriptors, or other
Python-visible protocols.

Lookup and insertion use separate probe opcodes so lookup does not carry
insertion-only tombstone state. Probe results are reserved SMIs:

- lookup returns `ProbeMiss`, `ProbeContinue`, or a non-negative entry index
- insertion returns `InsertProbeEmpty`, `InsertProbeTombstone`,
  `InsertProbeHashMiss`, or a non-negative entry index

The opcodes return indexes rather than pointers. No table pointer or entry
reference may survive a Python-visible call.

## General Probing And Reentrancy

General lookup computes the canonical hash, starts a probe, and examines only
entries with the same stored hash. Identity is tested before equality. When
identity fails, generated bytecode executes ordinary `==` dispatch and
`ToBool`.

Insertion uses the same identity/equality rules, records the first usable
tombstone, and distinguishes an existing equal key from a new key. Deletion and
`pop` retain any result value before invalidating the entry.

`get`, `__contains__`, `setdefault`, and `pop` distinguish an ordinary
miss from every exception raised by hash or equality. `setdefault` and `pop`
perform one probe rather than composing multiple public operations, avoiding
duplicate protocol calls and extra reentrancy points.

General hashing and equality can raise, mutate the same dictionary, insert or
remove colliding keys, resize or clear the table, and re-enter dict operations.

The implementation therefore:

- owns the input key and insertion value across protocol calls
- owns a candidate key across equality
- records the probe-structure generation
- revalidates the generation, probe slot, entry index, and candidate identity
  after equality
- restarts from the current table when revalidation fails

The generation describes probe-structure validity, not every dictionary
mutation. Promotion, growth, clearing, or replacement increments it. An
in-place value overwrite does not. Local entry and slot checks detect deletion
or replacement without forcing unrelated mutations to invalidate every probe.

This follows CPython's local table/key revalidation model rather than relying on
one broad public mutation version.

## Construction And Bulk Operations

`CreateDict` receives evaluated key/value pairs, allocates an exact builtin
dictionary, and inserts each pair semantically in source order. A non-string key
promotes the result naturally. Before insertion, the opcode publishes the
current frame frontier so nested Clover calls cannot overwrite registers holding
later pairs.

`dict.update` supports mappings, iterables of pairs, and keyword arguments.
Each contributed pair goes through semantic insertion. Updates from an exact
dict may reuse a source entry's canonical stored hash, but target probing and
equality remain semantic.

Exact builtin `dict(...)` construction is implemented by a private
Python-backed `__new__`. It applies positional input before keyword arguments.
There is no `dict.__init__` population path.

`clear`, `copy`, `keys`, `values`, `items`, `popitem`, and iteration
operate on either shape without promoting merely to inspect stored entries.
Iterators track dictionary size and raise `RuntimeError` when the size changes
during iteration. Shape-only promotion does not change size and does not by
itself invalidate an iterator.

## C API

The native extension API exposes semantic dictionary operations only. It does
not expose shapes or raw string-keyed storage. Arbitrary-key and UTF-8
string-key variants use the same promotion, hash normalization, equality,
revalidation, and pending-exception behavior as Python-visible dictionaries.

Lookup, setdefault, and pop use explicit found/presence outputs rather than
borrowed-reference or null-sentinel conventions. Full signatures and output
contracts are documented in [Clover C API](clover-c-api.md#dictionary-api).

## Performance Contract

The canonical exact-string path must remain protocol-free. Its focused
benchmarks are:

- `BM_GetItemDictStringKeyed`
- `BM_SetItemDictStringKeyed`

The promoted controls are `BM_GetItemDictGeneral` and
`BM_SetItemDictGeneral`. Interpreter handlers used by these paths must continue
to satisfy the repository's frameless/musttail hot-handler checks.

The general path is expected to be more expensive because it contains explicit
special-method and equality inline caches. That work must stay in bytecode
rather than being hidden in opaque C++ probing helpers.

## Current Limitations

- Native `dict` subtype construction and subtype-specific dictionary behavior
  are not implemented.
- `dict.fromkeys` accepts only tuple and list key sources and is not installed
  with proper classmethod binding. General iterable and subclass-aware
  construction require classmethod descriptor support.
- Exact builtin construction is implemented entirely by `dict.__new__`;
  `dict.__init__` cannot reinitialize an existing dictionary.
- Direct or uncached calls into generated methods can conservatively promote an
  exact-string dictionary even when the key is a string. This affects
  optimization state, not Python-visible mapping semantics.
